# Experiment Loop

**목적**: 코드 오토리서치 실행을 측정 가능하고, 되돌릴 수 있고, 학습 가능한 상태로 유지한다.

## 1. Baseline First

- 실험 `0`이 기록되기 전에는 대상 코드베이스를 절대 변이하지 않는다.
- eval 자체가 틀렸다는 증거가 나오지 않는 한 baseline과 후속 실험에서 같은 테스트 프롬프트와 eval 세트를 유지한다.
- eval 세트를 바꿔야 한다면 점수를 섞지 말고 별도의 reset 이벤트로 기록한다.

## 2. 수정 전에 실패를 진단한다

변이를 고르기 전에 지배적인 실패를 분류한다:

- 느린 빌드나 시작 시간
- 과도한 번들 또는 아티팩트 크기
- 느린 쿼리, 요청, 렌더 경로
- 중복 로직이나 dead code
- flaky test나 신뢰할 수 없는 검증
- 작은 안전 변경을 가로막는 구조 마찰
- 반복 작업에 드는 개발자 비용이 큼

가장 많은 점수를 잃게 하거나 가장 자주 나타나는 실패를 먼저 고른다.

## 3. 한 번에 하나만 바꾼다

좋은 변이:

- hot path 하나를 단순화한다
- 측정 가능한 병목에 캐시, 배치, 가드 하나를 추가한다
- 중복 브랜치나 죽은 의존성 하나를 제거한다
- 비싼 연산 하나를 critical path 밖으로 옮긴다
- 재작업을 막는 검증 단계 하나를 조인다
- 측정 가능한 부담만 만드는 설정이나 추상화 하나를 삭제한다

나쁜 변이:

- 고립된 가설 없이 대규모 재작성
- 무관한 편집 여러 개를 한 실험에 묶기
- 측정 근거 없이 도구나 의존성을 더하기
- 사용자가 실제로 중요하게 보지 않는 지표 개선

## 4. Keep or Discard

- 총점이 오르면 **KEEP**.
- 총점이 그대로인데 복잡성이 늘면 **DISCARD**.
- 총점이 내려가면 **DISCARD**.
- 점수는 같지만 코드베이스가 실질적으로 단순해졌다면, 단순화 근거와 무회귀 증거를 명시한 경우에만 유지한다.

## 5. 코드 최적화 규칙

코드베이스를 최적화할 때는:

- 새 레이어보다 삭제를 우선한다
- 광범위한 cleanup 전에 타겟 검증으로 동작을 지킨다
- 새 추상화보다 기존 코드베이스 패턴을 재사용한다
- 각 실험은 점수 변화 이유를 설명할 수 있을 만큼 작게 유지한다

## 6. 로깅 규칙

모든 실험은 다음을 기록해야 한다:

- 실험 번호
- 점수와 최대 점수
- 통과율
- keep 또는 discard
- 변이를 설명하는 한 문장
- 이 변이가 왜 도움이 될 것이라고 봤는지
- 실제 eval 결과가 무엇 때문에 달라졌는지

## 7. 종료 조건

다음 중 하나가 참이 되면 멈춘다:

- 사용자가 실행을 멈춘다
- 예산 상한에 도달한다
- 코드베이스가 세 번 연속 keep 실험에서 `95%+` 통과율을 기록한다
- 남은 실패가 코드베이스가 아니라 나쁜 eval 설계 때문이라고 판단된다

eval은 통과하지만 실제 결과가 여전히 약하면, 변이를 더하기 전에 eval을 먼저 고친다.
