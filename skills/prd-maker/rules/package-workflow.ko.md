# 패키지 워크플로

러프한 제품 요청을 전체 계층형 기획 패키지의 생성 또는 갱신 흐름으로 바꿀 때 이 규칙 파일을 사용합니다.

## 1. 모드 먼저 선택

- `create`: 이니셔티브와 일치하는 기존 패키지 폴더가 없거나, 사용자가 새 기획 패키지를 명시적으로 원함.
- `update`: 기존 패키지가 있고 요청이 범위, 요구사항, 스펙, 플로우, 와이어프레임, 메트릭, 리스크, 출시 세부, 검토 보조 산출물을 다듬음.

사용자가 이미 잡힌 이니셔티브를 반복 개선하는 상황이면 `update`를 우선합니다.

## 2. 최소 작업 브리프 수집

작성 전에 다음을 식별하거나 추론합니다.

- 제품 또는 기능명
- 해결할 문제
- 대상 사용자 또는 운영자
- 원하는 결과 또는 비즈니스 목표
- 주요 사용 사례
- 알려진 제약 또는 마감
- 예상 플랫폼 또는 표면
- 알려진 미지수

요청이 빈약하면 저위험 기본값만 추론하고 나머지는 명시적 가정 또는 오픈 질문으로 남깁니다.

## 3. 조사 필요 여부 결정

요구사항이 다음에 의존하면 라이브 조사를 수행합니다.

- 최신 시장 또는 경쟁사 주장
- 최근 제품, 사용자, 법무, 플랫폼, 기술 사실
- 사용자가 제공하지 않은 특정 메트릭, 벤치마크, 가격, 정책, API
- 안전, 컴플라이언스, 금융 관련 주장

조사 규칙:

- 서로 다른 각도의 distinct query를 사용합니다.
- 공식, 1차, 또는 권위 있는 출처를 우선합니다.
- 패키지 결정을 뒷받침하기에 근거가 충분하면 멈춥니다.
- 쿼리, 링크, 조회일, 근거 공백을 `sources.md`에 기록합니다.

사용자 맥락만으로 충분하고 남은 미지수를 안전하게 표시할 수 있으면 외부 조사를 생략합니다.

## 4. 의존 순서대로 작성

새 패키지에는 이 순서를 사용합니다.

1. 작업 맥락, 제공 제약, 조사 메모용 `sources.md`
2. 제품 결정 기록과 요구사항 ID를 담는 `prd.md`
3. PRD 요구사항에서 도출한 기능 동작 `feature-spec.md`
4. 사용자-facing 기능 동작에서 도출한 사용자 경로 `user-flow.md`
5. 유저플로우 단계와 상태에서 도출한 저충실도 화면 구조 `wireframe.md`
6. 기획 맵 wrapper인 `diagram.md`, `diagram.data.json`, 렌더링된 `diagram.svg`
7. 로컬 브라우저 미리보기 `preview.html`

이 순서는 PRD를 제품 기준으로 유지하고 후속 문서를 파생 검토 산출물로 만듭니다. 기능명세서, 유저플로우, 와이어프레임을 PRD와 분리된 독립 문서처럼 작성하지 않습니다.

## 5. 계층형 인수인계 계약 적용

다음 추적 체인을 사용합니다.

| 원천 산출물 | 반드시 넘길 내용 | 후속 산출물 |
|---|---|---|
| `prd.md` | 요구사항 ID, 목표, 범위, 비목표, 메트릭, 리스크, 오픈 질문 | `feature-spec.md` |
| `feature-spec.md` | 기능 요구사항 ID, 트리거, 상태, 권한, 오류, 수락 기준 | `user-flow.md` |
| `user-flow.md` | 액터, 진입점, 결정 지점, 정상/대안/오류 경로, 화면 상태 필요사항 | `wireframe.md` |
| `wireframe.md` | 화면 ID, 레이아웃 블록, 상태 변형, 미해결 디자인/제품 질문 | `diagram.md` 및 `preview.html` |

후속 산출물이 누락된 결정을 드러내면 상위 산출물을 갱신하거나 그곳에 명시적 오픈 질문을 추가합니다. 공백을 후속 파일에만 숨기지 않습니다.

## 6. 재작성 대신 갱신

갱신할 때는:

- 기존 패키지 파일을 먼저 모두 읽습니다.
- 여전히 유효한 결정은 보존합니다.
- 영향받은 섹션만 patch합니다.
- `prd.md`에 날짜 있는 변경 이력 행을 추가합니다.
- PRD 변경이 동작, 플로우, 화면에 영향을 주면 후속 문서를 갱신합니다.
- 새 근거는 `sources.md`에 추가하고, 명백히 낡은 경우가 아니면 이전 근거를 삭제하지 않습니다.
- 패키지 콘텐츠, 다이어그램 데이터, 출처 로그가 바뀌면 `diagram.svg`와 `preview.html`을 다시 생성합니다.

## 7. 산출물 정렬 유지

작성 또는 갱신 후 cross-document 일관성을 확인합니다.

- 모든 필수 PRD 요구사항이 `feature-spec.md`에 나타납니다.
- 모든 기능명세서 동작은 유저플로우 경로나 명시적 non-user-facing 메모를 가집니다.
- 모든 사용자-facing 플로우는 와이어프레임 화면 또는 화면 상태 항목을 가집니다.
- 다이어그램에는 중앙 이니셔티브 노드와 PRD, 기능명세서, 유저플로우, 와이어프레임, 오픈 질문 가지가 있습니다.
- `preview.html`은 최신 패키지 파일과 렌더링된 다이어그램을 반영합니다.
- research-backed non-obvious claim마다 출처 항목이 있습니다.
- 모든 미해결 모호성은 오픈 질문 또는 가정에 보입니다.
