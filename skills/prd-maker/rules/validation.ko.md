# 검증

기획 패키지 완료를 선언하기 전에 이 체크리스트를 사용합니다.

## 필수 확인

- 요청이 실제로 제품 기획 패키지, PRD, 기능명세서, 유저플로우, 저충실도 와이어프레임, 다이어그램, 미리보기를 요구했습니다.
- 패키지 폴더가 `.hypercore/prd/[slug]/` 아래에 있습니다.
- 새 패키지는 `prd.md`, `feature-spec.md`, `user-flow.md`, `wireframe.md`, `diagram.md`, `diagram.data.json`, `diagram.svg`, `preview.html`, `sources.md`를 포함합니다.
- PRD에는 목표, 범위, 비목표, ID가 있는 요구사항, 메트릭, 리스크/의존성, 오픈 질문, 변경 이력이 있습니다.
- 기능명세서는 모든 must-have PRD 요구사항을 기능 동작, 수락 기준, 상태, 권한, 오류, 분석, 필요 시 출시 메모로 다룹니다.
- 유저플로우는 액터, 진입점, 정상 경로, 대안 경로, 예외/오류 상태, 복구 경로, 종료 지점을 다룹니다.
- 와이어프레임은 각 사용자-facing 플로우를 화면 목록, 저충실도 레이아웃 블록, 컴포넌트, 상태 변형, 주석, 미해결 디자인/제품 질문으로 다룹니다.
- 다이어그램에는 중앙 아이디어 노드, PRD/spec/flow/wireframe 가지, 2차 노드, 오픈 공백, 유효한 Mermaid/text 구조, 유효한 JSON 노드 데이터, 렌더링된 SVG가 있습니다.
- 미리보기 HTML은 외부 의존성 없이 열리고 다이어그램과 텍스트 산출물을 포함합니다.
- research-backed non-obvious claim에는 `sources.md` 링크가 있습니다.
- `sources.md`에는 제공된 맥락과 서로 다른 쿼리 또는 외부 조사가 불필요했다는 명시 메모가 있습니다.
- 갱신 시 PRD 변경 이력은 무엇이 왜 바뀌었는지 보여줍니다.
- 미지수는 조용히 생략되지 않고 오픈 질문 또는 가정으로 보입니다.
- 복잡 패키지는 현재 단계 상태와 최종 검증 상태가 있는 `flow.json`을 포함합니다.

## 체인 추적성 확인

- 모든 must-have PRD 요구사항은 기능명세서 행 또는 명시적 defer 항목에 매핑됩니다.
- 모든 사용자-facing 기능명세서 동작은 유저플로우 단계, 결정, 또는 명시적 non-user-facing 메모에 매핑됩니다.
- 화면이 필요한 모든 유저플로우 단계는 와이어프레임 화면/상태 항목에 매핑됩니다.
- 모든 와이어프레임 화면은 목적, 관련 플로우 단계, 주요 액션 또는 상태 근거, 불확실할 때의 오픈 질문을 가집니다.
- 후속 산출물은 새 제품 결정을 만들 때 `prd.md`에 반영하거나 오픈 질문으로 표시합니다.

## 산출물 간 확인

- 다이어그램은 패키지를 한눈에 매핑하고 텍스트 산출물에 연결됩니다.
- PRD 요구사항은 기능명세서 요구사항에 매핑됩니다.
- 기능명세서 동작은 유저플로우 경로 또는 명시적 non-user-facing 메모에 매핑됩니다.
- 사용자-facing 플로우는 와이어프레임 화면 또는 화면 상태에 매핑됩니다.
- 원천 산출물이 바뀐 뒤 미리보기 콘텐츠를 다시 빌드했습니다.
- 메트릭과 분석 이벤트는 PRD와 기능명세서 사이에서 일관됩니다.
- 리스크, 의존성, 오픈 질문이 파일 간 모순되지 않습니다.

## 품질 확인

- 패키지는 빠르게 훑을 만큼 간결합니다.
- 주요 결정은 `prd.md`, 상세 동작은 `feature-spec.md`, 경로는 `user-flow.md`, 화면 구조는 `wireframe.md`, 근거 축적은 `sources.md`에 있습니다.
- 요구사항은 진짜 필요한 제약이 아닌 한 구현 명령이 아니라 제품 동작 또는 결과로 작성됩니다.
- 유저플로우는 happy-path 화면 순서만이 아니라 의미 있는 결정과 복구 경로를 모델링합니다.
- 와이어프레임은 이른 시각 polish가 아니라 저충실도 구조입니다.

## Forward-test 질문

- 새 팀원이 이 이니셔티브가 어떤 문제를 해결하는지 이해할 수 있는가?
- 엔지니어링이 필수, 선택, 범위 밖을 구분할 수 있는가?
- QA가 수락 기준과 플로우 예외 케이스에서 테스트 시나리오를 도출할 수 있는가?
- 디자인이 visual comp를 만들기 전에 필요한 화면과 상태를 이해할 수 있는가?
- 다음 달에 누군가 패키지를 다시 봐도 무엇이 바뀌었는지 알 수 있는가?
- 근거가 바뀌면 패키지 전체를 다시 쓰지 않고 `sources.md`를 갱신할 수 있는가?
