# PRD 워크플로

요청을 create 또는 update PRD 흐름으로 바꿀 때 이 규칙 파일을 사용합니다.

## 1. 먼저 모드를 고릅니다

- `create`: 이 이니셔티브와 맞는 PRD 폴더가 아직 없거나, 사용자가 새 PRD를 명시적으로 원함
- `update`: 기존 PRD가 있고 요청이 개선, 확장, 정정, 최신화임

이미 진행 중인 이니셔티브를 다루는 요청이라면 `update`를 우선합니다.

## 2. 최소 작업 브리프를 모읍니다

작성 전에 다음을 식별합니다.

- 제품 또는 기능 이름
- 해결할 문제
- 대상 사용자 또는 운영자
- 원하는 결과 또는 비즈니스 목표
- 현재 제약이나 마감
- 이미 드러난 미지수

이 정보가 비어 있으면 저위험 기본값만 추론하고, 나머지는 오픈 질문으로 남깁니다.

## 3. 조사 필요 여부를 결정합니다

PRD가 다음에 의존하면 라이브 조사를 수행합니다.

- 최신 시장 또는 경쟁사 주장
- 최근 제품/사용자 근거
- 날짜에 민감한 기술, 법무, 플랫폼 제약
- 사용자가 주지 않은 구체 숫자

조사 규칙:

- 쿼리는 서로 다른 각도로만 사용합니다.
- 1차 또는 공식 소스를 먼저 봅니다.
- PRD를 지지할 근거가 충분해지면 멈춥니다.
- 링크와 조회일은 `sources.md`에 기록합니다.

## 4. PRD를 작성합니다

메인 문서는 압축되고 의사결정 지향적으로 유지합니다.

항상 포함할 축:

- 개요와 상태
- 문제와 목표
- 사용자와 유스케이스
- 범위와 비목표
- 요구사항
- 메트릭
- 가정, 리스크, 의존성
- 오픈 질문
- 변경 이력

선택 섹션은 실제로 모호성을 줄일 때만 추가합니다.

## 5. 갈아엎기보다 갱신합니다

갱신 시에는:

- 아직 유효한 결정은 보존합니다
- 영향받는 섹션만 수정합니다
- 무엇이 왜 바뀌었는지 적습니다
- 해결되지 않은 충돌은 숨기지 않습니다

## 6. 연결된 상세는 본문 밖에 둡니다

- 원시 메모, 쿼리 로그, 소스 요약은 `sources.md`에 둡니다.
- 디자인, 티켓, 인터뷰, 외부 문서는 PRD에 링크만 걸고 전문 복사는 피합니다.
