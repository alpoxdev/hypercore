# 필수 행동

**목적**: docs-maker 생성/리팩토링 작업에서 반드시 따라야 하는 규칙입니다.

## 1. 작업 시작

### 필수 1: 주요 편집 전 계획 수립

중요한 생성 또는 리팩토링 작업 전에는 `sequential-thinking`을 사용합니다.

| 복잡도 | 최소 단계 | 대표 작업 |
|------|------|------|
| LOW | 1-2 | 짧은 문구 수정, 빠른 읽기 |
| MEDIUM | 3-5 | 문서 리팩토링, 규칙 정리 |
| HIGH | 5-8+ | 다중 파일 스킬 재설계, 하네스 구조 개편 |

### 필수 2: 수정 전 읽기

수정 전에 대상 파일을 읽고 다음 항목을 파악합니다.

- 유지해야 하는 핵심 의도
- 통합할 수 있는 반복 규칙
- 제거해야 하는 오래된 가이드
- 분리해야 하는 공급자 민감한 내용

### 필수 3: 독립 파일은 병렬 읽기

현재 규칙 집합을 이해하기 위해 3개 이상 독립 파일이 필요하면 병렬로 읽습니다.

## 2. 문서 작성

### 필수 4: 의도는 보존하고 형태는 개선

리팩토링 모드에서는:

- 오래되었거나 더 강한 근거와 충돌하지 않는 한 기존 운영 의도를 유지합니다
- 명확성, 밀도, 검증 구조를 개선합니다
- 혼합 관심사와 중복 내용을 제거합니다

### 필수 5: 규칙을 명시적이고 테스트 가능하게 작성

중요한 규칙은 다음에 답해야 합니다.

- 무엇을 해야 하는가
- 언제 적용되는가
- 어떻게 검증하는가

### 필수 6: 안정적인 구조 사용

검색 신뢰성을 높일 수 있다면 명시적 섹션, 표, 압축된 체크리스트를 우선합니다.

### 필수 7: 규칙에 예시를 붙이기

규칙이 자명하지 않다면 복사 가능한 예시나 패턴을 함께 둡니다.

### 필수 8: 개념당 용어 하나

문서 안에서 같은 개념을 여러 이름으로 부르지 않습니다. 구분이 필요하면 그 차이를 먼저 설명합니다.

## 3. 하네스 문서화

### 필수 9: 코어 규칙과 공급자 민감 규칙 분리

다음 구조를 사용합니다.

- 표준 코어: 오래 살아남는 공급자 중립 가이드
- 공급자 참조 또는 어댑터: 날짜가 붙은 공급자 세부사항
- 로컬 선호: 프로젝트별 관례

### 필수 10: 표준 문서에서 고정 모델명 제거

표준 skill과 rule 파일에서는:

- `frontier reasoning model`, `snapshot-pinned production model` 같은 성능 프로필을 우선합니다
- 실제 모델 문자열은 공급자 참조, migration 노트, 배포 예시로 이동합니다

### 필수 11: 공급자 민감 주장에 출처 부착

Anthropic 또는 OpenAI 동작에 기대는 규칙은 다음 필드를 가진 참조 항목으로 연결합니다.

- `source_url`
- `last_verified_at`
- `applies_to`
- `summary`
- `implication_for_docs_maker`

### 필수 12: 프롬프트만이 아니라 전체 하네스를 문서화

범위가 하네스 엔지니어링이면 다음을 명시적으로 다룹니다.

- 프롬프트 자산과 템플릿
- 도구 계약과 설명
- 평가 데이터셋과 평가기
- 안전과 승인 게이트
- 컨텍스트 배치, 상태, 압축
- 모델과 버전 프로필 전략

## 4. 검증

### 필수 13: 혼합 관심사 정리 여부 검증

리팩토링 후 docs-maker 기본 표면에 무관한 구현 스택 규칙이나 프로젝트 전용 워크플로가 남아 있지 않은지 확인합니다.

### 필수 14: 출처 최신성 검증

공급자 민감한 지시는 인용된 공식 문서가 현재 주장에 충분히 최신인지 확인합니다.

### 필수 15: 문구와 grep 위생 검증

필요할 때 저장소 검색으로 다음을 확인합니다.

- 오래된 고정 모델명이 표준 문서에서 제거되었는지
- 중복 문구가 정리되었는지
- 이동한 규칙이 예전 파일에 남아 있지 않은지

### 필수 16: 리뷰어 실패 기준 적용

완료 선언 전 다음 조건이 모두 거짓인지 확인합니다.

- 표준 문서가 코어 규칙과 공급자 세부사항을 경계 없이 섞고 있음
- 참조가 존재하지만 실제 규칙을 충분히 뒷받침하지 못함
- 하네스 문서인데 도구, 평가, 안전, 컨텍스트 경계가 빠져 있음
- 예시가 규칙보다 더 구체적이라 로컬 선호를 몰래 주입함

### 필수 17: 변경 내용 보고

완료 보고에는 다음을 포함합니다.

- 변경 파일
- 단순화한 내용
- 남은 리스크 또는 후속 작업
