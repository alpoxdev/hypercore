# Plan Command

개발 진행 방법을 검토하고 2-3개 옵션과 장단점을 제시하는 커맨드.

## CRITICAL: 필수 요구사항

| 요구사항 | 설명 |
|----------|------|
| **ARGUMENT 필수** | 계획 대상 없이 실행 시 되물음 |
| **Ultrathink** | 깊은 사고로 옵션 분석 |
| **Sequential Thinking 3-7** | 복잡도에 따라 조정 |
| **코드 수정 금지** | 분석과 계획만. Edit 사용 불가 |
| **옵션 2-3개** | 최소 2개, 권장 3개 |

## ARGUMENT 확인

```
$ARGUMENTS 없음 → 즉시 질문:
"무엇을 계획해야 하나요? 구체적으로 알려주세요.
- 새 기능 추가
- 버그 수정
- 리팩토링
- 아키텍처 변경"

$ARGUMENTS 있음 → 다음 단계 진행
```

**계획 대상**: $ARGUMENTS

## 실행 흐름

```
1. ARGUMENT 확인 (없으면 되물음)
2. Sequential Thinking으로 복잡도 판단 (1단계)
3. 복잡도에 따른 분석 (2-6단계)
   - 현재 상태 파악
   - 가능한 접근 방식 탐색
   - 옵션별 장단점 분석
4. Task subagent로 코드베이스 탐색
5. 옵션 2-3개 정리 및 제시
6. 사용자 선택 대기
7. 문서 작성 필요 여부 질문
   - Y → 계획 문서 작성
   - N → 종료
```

## 복잡도 판단

| 복잡도 | 사고 횟수 | 예시 |
|--------|----------|------|
| 간단 | 3 | 단일 파일 수정, 명확한 변경 |
| 보통 | 5 | 다중 파일, 로직 변경 |
| 복잡 | 7+ | 아키텍처 변경, 새 시스템 설계 |

**복잡도 판단 기준:**

```
thought 1: 복잡도 판단
- 영향 범위: 파일 수, 모듈 수
- 의존성: 외부 라이브러리, 다른 시스템
- 리스크: 기존 기능 영향, 롤백 가능성
- 기술적 난이도: 새로운 패턴, 미지의 영역
```

## Sequential Thinking 가이드

**기본 흐름 (5단계):**

```
thought 1: 복잡도 판단 및 분석 범위 결정
thought 2: 현재 상태 분석 (코드, 아키텍처, 제약)
thought 3: 가능한 접근 방식 열거 (4-5개)
thought 4: 주요 옵션 3개 선정 및 장단점 분석
thought 5: 최종 옵션 정리 및 추천안 도출
```

**복잡한 경우 (7단계):**

```
thought 1: 복잡도 판단
thought 2: 현재 상태 심층 분석
thought 3: 기술적 제약 및 요구사항 정리
thought 4: 가능한 접근 방식 탐색
thought 5: 각 접근 방식 비교 분석
thought 6: 옵션 3개 선정 및 상세 장단점
thought 7: 추천안 및 근거
```

## Task Subagent 활용

| subagent_type | 용도 |
|---------------|------|
| `Explore` | 코드베이스 구조 파악, 관련 파일 탐색 |
| `Plan` | 구현 전략 수립, 단계별 계획 |
| `general-purpose` | 복잡한 분석, 다중 시스템 연관 |

**Task 호출 예시:**

```
Task: "새 인증 시스템 구현을 위한 코드베이스 분석"
subagent_type: Explore
prompt: "현재 인증 관련 코드 구조 파악.
        사용 중인 라이브러리, 세션 관리 방식,
        수정이 필요한 파일 목록 제시."
```

**병렬 탐색:**

```
여러 측면 분석 필요 시:
Task 1: "프론트엔드 구조 분석" (Explore)
Task 2: "백엔드 API 분석" (Explore)
Task 3: "데이터베이스 스키마 분석" (Explore)

→ 단일 메시지에 다중 Task 호출
→ 결과 취합 후 옵션 정리
```

## 옵션 제시 형식

**옵션 3개 제시:**

```
## 분석 결과

### 옵션 1: [옵션 이름] (추천)

**접근 방식:**
- 설명 1
- 설명 2

| 장점 | 단점 |
|------|------|
| 장점 1 | 단점 1 |
| 장점 2 | 단점 2 |

**영향 범위:**
- 파일: `src/auth/`, `src/api/`
- 예상 변경 규모: 중간

---

### 옵션 2: [옵션 이름]

**접근 방식:**
...

---

### 옵션 3: [옵션 이름]

**접근 방식:**
...

---

## 추천 및 근거

옵션 1을 추천합니다.
- 근거 1
- 근거 2

어떤 옵션을 선택하시겠습니까? (1/2/3)
```

**옵션 2개 제시 (선택지가 명확한 경우):**

```
## 분석 결과

두 가지 접근 방식이 있습니다:

### 옵션 A: [옵션 이름]
...

### 옵션 B: [옵션 이름]
...

어떤 옵션을 선택하시겠습니까? (A/B)
```

## 문서 작성 질문

**옵션 선택 후 질문:**

```
옵션 [N]을 선택하셨습니다.

계획 문서를 작성할까요?
- Y: .claude/plans/[기능명].md 생성
- N: 바로 구현 시작

선택해주세요. (Y/N)
```

## 계획 문서 템플릿

**파일 위치:** `.claude/plans/[기능명].md`

```markdown
# [기능명] 구현 계획

## 개요

**목표:** [무엇을 달성할 것인가]
**선택된 접근 방식:** [옵션 N]
**예상 영향 범위:** [파일/모듈 목록]

## 현재 상태

- 현재 구조 설명
- 관련 코드 위치
- 기존 제약사항

## 구현 단계

### 1단계: [단계 이름]

**작업:**
- [ ] 작업 1
- [ ] 작업 2

**변경 파일:**
- `src/file1.ts`
- `src/file2.ts`

### 2단계: [단계 이름]
...

## 고려사항

### 리스크

| 리스크 | 완화 방안 |
|--------|----------|
| 리스크 1 | 방안 1 |

### 의존성

- 외부 라이브러리: [목록]
- 다른 시스템: [목록]

### 롤백 계획

문제 발생 시 롤백 방법.

## 검증 방법

- 테스트 항목 1
- 테스트 항목 2

## 참조

- 관련 문서 링크
- 참고 자료
```

## 예시 워크플로우

```
1. 사용자: /plan 사용자 인증을 JWT에서 세션 기반으로 변경

2. Sequential Thinking:
   thought 1: "인증 시스템 변경 - 복잡도 높음, 7단계 분석"
   thought 2: "현재 JWT 구현 분석 필요..."
   ...

3. Task 탐색:
   Task: "현재 인증 구현 분석" (Explore)

4. 옵션 제시:
   옵션 1: 점진적 마이그레이션 (추천)
   옵션 2: 완전 대체
   옵션 3: 하이브리드 방식

5. 사용자 선택: 1

6. 문서 작성 질문: Y

7. .claude/plans/session-auth.md 생성
```

## CRITICAL: 절대 금지

| 금지 항목 |
|----------|
| ARGUMENT 없이 분석 시작 |
| Edit 도구 사용 (코드 수정) |
| Sequential Thinking 3단계 미만 |
| 옵션 1개만 제시 |
| 코드 탐색 없이 추측으로 옵션 제시 |
| 사용자 선택 없이 구현 시작 |
| 장단점 없이 옵션만 나열 |
