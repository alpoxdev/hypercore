# 검증

기획 패키지가 끝났다고 말하기 전에 이 체크리스트를 사용합니다.

## 필수 체크

- 요청이 실제 제품 기획 패키지, PRD, 다이어그램, 기능 명세, 유저 플로우, 와이어프레임을 요구했다.
- 패키지 폴더가 `.hypercore/prd/[slug]/` 아래에 존재한다.
- 새 패키지에는 `prd.md`, `diagram.md`, `diagram.data.json`, `diagram.svg`, `feature-spec.md`, `user-flow.md`, `wireframe.md`, `preview.html`, `sources.md`가 있다.
- PRD에 목표, 범위, 비목표, 요구사항, 메트릭, 리스크/의존성, 오픈 질문, 변경 이력이 있다.
- 다이어그램이 중앙 아이디어 노드, PRD/스펙/플로우/와이어프레임 가지, 2차 노드, 오픈 공백, 유효한 Mermaid/text 구조, 유효한 JSON 노드 데이터, 렌더링된 SVG를 포함한다.
- 기능 명세가 모든 필수 요구사항을 수락 기준, 상태, 권한, 오류, 분석 이벤트, 필요 시 출시 메모와 함께 다룬다.
- 유저 플로우가 액터, 진입점, 정상 경로, 대안 경로, 예외/오류 상태, 종료 지점을 다룬다.
- HTML 미리보기가 외부 의존성 없이 열리고 다이어그램과 텍스트 산출물을 포함한다.
- 와이어프레임이 각 사용자-facing 플로우를 화면 목록, 레이아웃 블록, 컴포넌트, 미해결 디자인/제품 질문으로 다룬다.
- 조사 기반 비자명한 주장은 `sources.md`에 링크가 있다.
- `sources.md`에는 제공된 맥락과 서로 다른 쿼리 또는 외부 조사가 불필요했다는 명시적 기록이 있다.
- 갱신한 경우 PRD 변경 이력에 무엇이 왜 바뀌었는지 드러난다.
- 미지수는 조용히 빠지지 않고 오픈 질문 또는 가정으로 남아 있다.
- 복잡 패키지는 현재 단계 상태와 최종 검증 상태를 담은 `flow.json`이 있다.

## 문서 간 체크

- 다이어그램이 패키지를 한눈에 매핑하고 텍스트 산출물로 연결된다.
- PRD 요구사항이 기능 명세 요구사항으로 이어진다.
- 기능 명세 동작은 유저 플로우 경로나 명시적 비사용자-facing 메모로 이어진다.
- 사용자-facing 플로우는 와이어프레임 화면 또는 화면 상태로 이어진다.
- 원본 산출물이 바뀐 뒤 미리보기 콘텐츠가 다시 생성되었다.
- 메트릭과 분석 이벤트가 PRD와 기능 명세 사이에서 일관된다.
- 리스크, 의존성, 오픈 질문이 파일 간 모순되지 않는다.

## 품질 체크

- 패키지가 빠르게 훑어볼 수 있을 만큼 압축되어 있다.
- 주요 결정은 `prd.md`, 상세 동작은 `feature-spec.md`, 근거 누적은 `sources.md`에 있다.
- 요구사항은 정말 필요한 제약이 아닌 한 구현 지시가 아니라 제품 동작 또는 결과 중심으로 적혀 있다.
- 와이어프레임은 성급한 시각 장식이 아니라 저충실도 구조다.

## 전방 테스트 질문

- 새 팀원이 이 이니셔티브가 어떤 문제를 푸는지 이해할 수 있는가?

- 디자인이 시각 컴프 전에 필요한 화면과 상태를 이해할 수 있는가?
- 한 달 뒤 다시 봤을 때 무엇이 바뀌었는지 알 수 있는가?
- 근거가 바뀌면 패키지 전체를 갈아엎지 않고 `sources.md`를 갱신할 수 있는가?
