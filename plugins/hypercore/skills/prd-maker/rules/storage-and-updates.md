# Storage And Updates

Use this rule file whenever the skill creates or updates package files.

## 1. Folder location

Store each planning package here:

`.hypercore/prd/[slug]/`

Default files for a new package:

- `prd.md`
- `diagram.md`
- `diagram.data.json`
- `diagram.svg`
- `feature-spec.md`
- `user-flow.md`
- `wireframe.md`
- `preview.html`
- `sources.md`

Complex packages also include:

- `flow.json`

## 2. Slug rules

- Prefer short ASCII kebab-case slugs.
- Base the slug on the initiative or feature, not the full request sentence.
- Reuse an existing slug when the user is clearly updating the same topic.

Good:

- `billing-retry-flow`
- `team-inbox-assignment`
- `ai-planning-assistant`

Avoid:

- full sentence slugs
- dated slugs unless the date is part of the product concept
- multiple folders for the same initiative

## 3. File responsibilities

- `prd.md`: product decision record and requirements
- `diagram.md`: visual planning map explanation and Mermaid/text source
- `diagram.data.json`: node data for deterministic SVG rendering
- `diagram.svg`: rendered card-and-connector visual map generated by `scripts/render-planning-map.mjs`
- `feature-spec.md`: functional behavior, acceptance criteria, states, permissions, analytics, rollout notes
- `user-flow.md`: user journeys, branches, decision points, empty/error states, entry/exit points
- `wireframe.md`: low-fidelity screen inventory and text wireframes
- `preview.html`: local browser viewer generated by `scripts/build-preview.mjs` from `assets/preview.template.html`
- `sources.md`: provided context, research queries, source notes, evidence gaps
- `flow.json`: phase state for complex packages only

Do not create extra README, notes, or changelog files unless the user explicitly asks for them.

## 4. Create flow

When the folder does not exist:

- create the folder
- create each markdown file from its matching template asset and create `diagram.data.json` from the JSON template
- fill unknowns with explicit assumptions or open questions, not silent omissions
- render `diagram.svg` from `diagram.data.json` with `scripts/render-planning-map.mjs`
- build `preview.html` with `scripts/build-preview.mjs` after package content is written
- if complex, initialize `flow.json` from `references/flow-schema.md`

## 5. Update flow

When the folder already exists:

- read current package files first
- add any missing default files from templates before updating dependent content
- patch only sections that changed
- preserve existing links, prior decisions, and useful context
- append a dated entry to the change-history section in `prd.md`
- append new research/context to `sources.md` without destroying earlier evidence unless clearly obsolete
- rebuild `preview.html` after any package content, diagram, or source-log change

## 6. Missing-folder recovery

If the user asks to update a package but no matching folder exists:

- create a new folder
- note the missing baseline as an assumption in `prd.md` change history
- continue in `create` mode
