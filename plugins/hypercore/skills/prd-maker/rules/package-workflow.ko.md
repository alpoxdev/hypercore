# 패키지 워크플로

러프한 제품 요청을 전체 기획 패키지의 create 또는 update 흐름으로 바꿀 때 이 규칙 파일을 사용합니다.

## 1. 먼저 모드를 고릅니다

- `create`: 이 이니셔티브와 맞는 기존 패키지 폴더가 없거나, 사용자가 새 기획 패키지를 명시적으로 원함
- `update`: 기존 패키지가 있고 범위, 요구사항, 다이어그램, 플로우, 와이어프레임, 메트릭, 리스크, 출시 세부사항을 개선하는 요청임

이미 진행 중인 이니셔티브를 다루는 요청이라면 `update`를 우선합니다.

## 2. 최소 작업 브리프를 모읍니다

작성 전에 다음을 식별하거나 추론합니다.

- 제품 또는 기능 이름
- 해결할 문제
- 대상 사용자 또는 운영자
- 원하는 결과 또는 비즈니스 목표
- 주요 유스케이스
- 알려진 제약이나 마감
- 예상 플랫폼 또는 화면
- 알려진 미지수

요청이 sparse하면 저위험 기본값만 추론하고 나머지는 명시적 가정 또는 오픈 질문으로 남깁니다.

## 3. 조사 필요 여부를 결정합니다

요구사항이 다음에 의존하면 라이브 조사를 수행합니다.

- 최신 시장 또는 경쟁사 주장
- 최근 제품, 사용자, 법무, 플랫폼, 기술 사실
- 사용자가 제공하지 않은 구체 메트릭, 벤치마크, 가격, 정책, API
- 안전, 컴플라이언스, 금융 관련 주장

조사 규칙:

- 서로 다른 각도의 쿼리를 사용합니다.
- 공식, 1차, 권위 있는 출처를 우선합니다.
- 패키지 결정을 뒷받침하기에 충분하면 멈춥니다.
- 쿼리, 링크, 조회일, 근거 공백을 `sources.md`에 기록합니다.

사용자 맥락이 충분하고 남은 미지수를 안전하게 표시할 수 있으면 외부 조사를 생략합니다.

## 4. 의존 순서대로 작성합니다

새 패키지는 다음 순서로 작성합니다.

1. `sources.md` 작업 맥락과 조사 메모
2. `prd.md` 제품 결정 기록
3. `diagram.md` 가지형 기획 맵
4. `feature-spec.md` 기능 상세와 수락 기준
5. `user-flow.md` 여정, 분기, 상태
6. `wireframe.md` 화면 목록과 저충실도 레이아웃 블록
7. `preview.html` 로컬 브라우저 미리보기

이 순서는 PRD를 제품 진실의 기준으로 두고, 하위 문서를 파생 검토 산출물로 유지합니다.

## 5. 갈아엎기보다 갱신합니다

갱신 시에는:

- 기존 패키지 파일을 모두 먼저 읽습니다
- 아직 유효한 결정은 보존합니다
- 영향받는 섹션만 수정합니다
- `prd.md`에 날짜가 있는 변경 이력 행을 추가합니다
- PRD 변경이 동작, 플로우, 화면에 영향을 주면 하위 문서도 갱신합니다
- 명확히 폐기된 경우가 아니면 기존 근거를 삭제하지 않고 `sources.md`에 새 근거를 추가합니다

## 6. 산출물을 정렬합니다

작성 또는 갱신 후 문서 간 일관성을 확인합니다.

- 다이어그램에는 중앙 이니셔티브 노드와 PRD, 기능 명세, 유저 플로우, 와이어프레임, 오픈 질문 가지가 있습니다
- 모든 필수 PRD 요구사항이 `feature-spec.md`에 나타납니다
- 모든 주요 기능 동작은 유저 플로우 경로나 명시적 비사용자-facing 메모가 있습니다
- 모든 사용자-facing 플로우는 와이어프레임 화면 또는 화면 상태 항목이 있습니다
- `preview.html`이 최신 패키지 파일과 렌더링된 다이어그램을 반영합니다
- 조사 기반 비자명한 주장은 source entry가 있습니다
- 모든 미해결 모호성은 오픈 질문 또는 가정에 드러납니다
