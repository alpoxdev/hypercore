# Drizzle Official Docs Summary

Verified on: 2026-06-28

Use this reference when Hono architecture decisions depend on Drizzle ORM, `drizzle-kit`, runtime-specific database drivers, schema files, migrations, or transactions.

## Confirmed points from official docs

1. Drizzle provides a database connection hub with runtime/dialect-specific pages for PostgreSQL, Neon, Cloudflare D1, Turso/libSQL, SQLite, PGLite, Bun SQL, and other drivers. Pick the driver that matches the deployment runtime.
Source: [Database connection](https://orm.drizzle.team/docs/connect-overview)

2. Drizzle schema is declared in TypeScript files. Drizzle Kit reads schema files or folders, so schema declarations should live in a stable, exported location.
Source: [Schema declaration](https://orm.drizzle.team/docs/sql-schema-declaration), [Drizzle config file](https://orm.drizzle.team/docs/drizzle-config-file)

3. `drizzle.config.ts` uses `schema` for the schema file/folder path and `out` for the migration output folder. The default migration output folder is `./drizzle`.
Source: [Drizzle config file](https://orm.drizzle.team/docs/drizzle-config-file), [`drizzle-kit generate`](https://orm.drizzle.team/docs/drizzle-kit-generate)

4. `drizzle-kit generate` creates SQL migrations from Drizzle schema changes. Generated migrations can then be applied with `drizzle-kit migrate`, Drizzle ORM's migration APIs, external tools, or direct SQL execution.
Source: [`drizzle-kit generate`](https://orm.drizzle.team/docs/drizzle-kit-generate), [Migrations](https://orm.drizzle.team/docs/migrations)

5. `drizzle-kit migrate` applies SQL migrations generated by `drizzle-kit generate`, reads the migration folder, checks the migrations log table, runs unapplied SQL, and logs applied migrations.
Source: [`drizzle-kit migrate`](https://orm.drizzle.team/docs/drizzle-kit-migrate)

6. Drizzle's migration docs distinguish database-first and codebase-first workflows. `pull`, `generate`, `push`, `migrate`, and `export` exist for different workflow needs, so a skill should not force one migration workflow unless the repo already standardizes on it.
Source: [Migrations](https://orm.drizzle.team/docs/migrations)

7. Drizzle's serverless performance guidance recommends declaring database connections and prepared statements outside handler scope when the runtime can reuse the process.
Source: [Serverless performance](https://orm.drizzle.team/docs/perf-serverless)

8. Cloudflare D1 support uses the D1 Worker binding with `drizzle-orm/d1`; the D1 docs show `drizzle(env.<BINDING_NAME>)` and Wrangler migration configuration with `migrations_dir`.
Source: [Cloudflare D1](https://orm.drizzle.team/docs/connect-cloudflare-d1)

9. Neon support has separate HTTP, WebSocket, and serverless driver paths. The docs state that HTTP is faster for single, non-interactive transactions, while WebSockets are needed for session or interactive transaction support.
Source: [Neon](https://orm.drizzle.team/docs/connect-neon)

10. Drizzle transactions use `db.transaction(async (tx) => ...)`. The docs also show returning values, rollback, nested transactions with savepoints, relational queries inside transactions, and dialect-specific transaction config.
Source: [Transactions](https://orm.drizzle.team/docs/transactions)

## How this affects the skill

- Hypercore Hono rules should treat Drizzle setup as runtime-specific rather than one universal import path.
- Database client construction belongs in a database/platform boundary, not in each route handler.
- Drizzle schema and migration folders must be discoverable from `drizzle.config.ts`.
- Migration generation/application is operational behavior and should not be silently performed as part of ordinary route edits.
- Transaction boundaries belong at the service/use-case layer; repository helpers that participate in a transaction should accept a `tx` or database dependency.
- Public API DTOs, OpenAPI schemas, and typed RPC response shapes should not be assumed to equal Drizzle row shapes.

## Version and docs caveats

- The Drizzle docs are live and version-sensitive. Re-check exact package versions and import paths against the installed `drizzle-orm` / `drizzle-kit` release before making code changes.
- The rule that repository helpers should accept `tx` is an architectural inference from the Drizzle transaction API shape, not a verbatim official requirement.
- Runtime pages for Neon, D1, Turso/libSQL, and other providers can change as provider SDKs change.
